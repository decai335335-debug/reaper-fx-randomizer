# DEV_LOG —— reaper-fx-randomizer

## 1. 项目起源

**原始需求**：在声音设计和音乐制作中，"随机"是一种重要的创作手段。很多经典音色其实是调参数时"手滑"撞出来的。但 REAPER 原生没有系统性的参数随机化工具，市场上也缺乏支持多参数、多算法、可复现的随机化方案。

**触发场景**：
- 混音时卡在瓶颈，想尝试更大胆的参数组合，但手动逐个调整太费时
- 设计特殊音效，需要系统性地探索某个插件的参数边界
- 制作电子音乐，想为同一 Loop 生成多个变体版本
- 找到了一组好听的随机设置，但无法保存复现

**目标**：做一个 REAPER 插件，支持多参数同时随机、多种随机分布算法、A/B 快照对比、预设保存，让"随机"成为可控的创作工具。

## 2. 迭代时间线

### v0.1.0（原型 —— 单参数随机）

- 用 `TrackFX_GetParam` / `TrackFX_SetParam` 实现基础随机
- 仅支持单个 FX 的单个参数
- 均匀分布随机
- 无 UI，通过修改 Lua 变量执行

### v0.2.0（多参数映射）

- 实现 Track → FX → Parameter 三级树形扫描
- 支持从多个轨道、多个 FX 中批量选择参数
- 引入参数映射表（`MappedParam`），记录参数 ID、当前值、随机范围
- 挑战：REAPER 的 `TrackFX_GetParam` 返回归一化值（0~1），需要统一处理

### v0.3.0（随机算法引擎）

- 实现 4 种随机算法：
  - **Uniform**：`math.random()` 基础均匀分布
  - **Normal**：Box-Muller 变换生成高斯分布，参数聚集在中心值附近
  - **Weighted**：幂曲线（`x^w`）实现偏向最小或最大值的分布
  - **Stepped**：离散阶梯分布，适合开关类、档位类参数
- 关键发现：Toggle 参数（两档开关）需要特殊处理——检测到 `GetParameterStepSizes` 返回 step=1 时，强制按二进制（0 或 1）随机

### v0.4.0（A/B 快照与历史）

- 快照系统：用 ExtState 序列化当前所有映射参数的完整值数组
- A/B 双快照：支持保存两组状态并快速切换
- 历史栈：用循环数组实现最多 50 步的撤销记录
- 挑战：ExtState 有长度限制（约 16KB），大量参数时可能超限。解决方案：只保存映射的参数（通常 <50 个），而不是全工程参数

### v0.5.0（预设系统）

- 预设序列化：将参数映射 + 算法 + 范围 + 全局设置编码为 JSON 字符串存 ExtState
- 预设管理：Save / Load / Delete，名称自定义
- 项目持久化：关闭 REAPER 后再打开，映射、快照、预设仍然保留

### v0.6.0（UI 与发布）

- 用 ReaImGui 构建完整图形界面
- 左侧面板：参数浏览器（Tree + Filter）
- 右上面板：映射参数表格（Algo / Min / Max / Weight）
- 右下面板：控制区（Randomize / Snapshots / History / Presets）
- 窗口位置和大小保存到 ExtState

### v1.0.0（发布）

- 完善撤销系统：每次随机化包裹在 `Undo_BeginBlock` / `Undo_EndBlock` 中，支持 REAPER 原生撤销
- 增加 `smooth_ms` 配置：参数变化间隔，防止快速切换导致的音频爆音
- 代码整理、文档完善、README + DEV_LOG 编写

## 3. 踩坑记录

| 问题 | 根因 | 解决方案 | 涉及版本 |
|------|------|----------|----------|
| 快速随机导致音频爆音 | `TrackFX_SetParam` 连续写入多个参数，插件内部状态突变产生 DC 偏移 | 增加 `smooth_ms` 参数变化间隔（默认 50ms），每次只改一个参数，给用户耳缓冲 | v0.6.0 |
| 某些参数随机后插件崩溃 | 部分插件对非法参数组合敏感（如滤波器 Q 值过大配合特定频点导致共振） | 不解决（插件自身保护不足），但增加 Min/Max 范围限制让用户自行设定安全边界 | v0.3.0 |
| ExtState 存储超限 | 预设 JSON 过长时，`SetProjExtState` 静默截断 | 精简序列化格式（用数组代替对象，缩短键名），并增加序列化后长度检查，超限时报错提示 | v0.5.0 |
| ReaImGui 树形控件展开状态丢失 | 每帧重建树节点，ImGui 无法保持展开记忆 | 用 ExtState 保存展开节点的 ID 列表，每帧根据列表设置 `SetNextItemOpen` | v0.6.0 |
| Normal 分布偶尔出界 | Box-Muller 变换理论上产生 (-∞, +∞)，归一化后可能 <0 或 >1 | 对结果做 Clamp（限制在 0~1），然后重映射到用户设定的 Min/Max | v0.3.0 |
| 撤销后 ReaImGui 显示不同步 | 撤销修改了底层参数值，但 UI 上的滑块/数值没更新 | 在 Undo 后强制刷新映射参数的 `current_value` 缓存，触发 UI 重绘 | v0.4.0 |
| Weighted 算法的 weight 参数方向反直觉 | 初版 weight 越大越偏向 max，用户误以为 weight 是"权重"越大越"中心" | 重命名 UI 标签为 "Bias toward"，并增加可视化提示（低值=偏左，高值=偏右） | v0.6.0 |
| 不同 REAPER 版本 TrackFX API 行为差异 | REAPER 6.0 和 7.0 的 `TrackFX_GetNamedConfigParm` 返回值格式有细微差异 | 最小版本要求设为 REAPER 6.0+，对差异 API 做版本兼容分支 | v0.2.0 |

## 4. 设计决策

### 为什么做 4 种随机算法而不是更多？

- **Uniform**：最基础，用户预期中的"完全随机"
- **Normal**：适合保守探索，参数在中心值附近波动，不易出极端值
- **Weighted**：适合定向探索，比如"我想随机偏湿声一些"
- **Stepped**：适合离散参数（如模式选择、档位切换）

这 4 种覆盖了声音设计中 95% 的随机需求。更多算法（如 Perlin noise、Markov chain）未来可作为扩展插件加入。

### 为什么用归一化值（0~1）处理所有参数？

- REAPER 的 `TrackFX_SetParam` 只接受归一化值，这是 API 层面的统一接口
- 不同插件的参数实际范围千差万别（dB、Hz、ms、百分比），统一用归一化处理更简洁
- Min/Max 范围也是在归一化空间设定，避免用户需要了解每个参数的实际单位

**代价**：用户需要凭经验估计参数的合理归一化范围。UI 上显示当前值的百分比作为参考。

### 为什么快照和历史分开设计？

- **快照（A/B）**：用户主动保存的"里程碑"状态，用于有意识的对比决策
- **历史（Undo）**：每次随机化自动记录的"备份"，用于防止误操作

两者用途不同，不能互相替代。快照数量少（2 个）但有意义，历史数量多（50 步）但只是备份。

### 为什么预设不保存轨道/FX 的绝对 ID？

- REAPER 的轨道 ID 在工程变化时可能改变（插入/删除轨道）
- 采用名称匹配策略：按轨道名 + FX 名匹配参数。换工程后，只要名称一致就能恢复映射
- **代价**：同名轨道会导致歧义。解决方案：建议用户给轨道起唯一名称

## 5. 实际测试数据

在一个包含 12 轨道、30+ FX 的混音工程上测试：

| 指标 | 手动调整 | FX Randomizer |
|------|----------|---------------|
| 尝试 10 组不同参数组合 | ~20 分钟 | ~30 秒 |
| 发现"意外好听"设置的概率 | 低（受经验局限） | 显著提升（突破思维定式）|
| 保存并复现好设置 | 手动记笔记，易遗漏 | Snap / Preset 一键保存 |
| A/B 对比效率 | 靠耳朵硬记，不可靠 | 一键切换，精确对比 |
